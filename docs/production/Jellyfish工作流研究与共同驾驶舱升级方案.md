# Jellyfish 工作流研究与共同驾驶舱升级方案

日期：2026-06-03

## 结论

Jellyfish 值得参考，但不建议整套迁移。

它最有价值的不是界面，也不是某个模型接口，而是把 AI 短剧制作拆成可管理的生产状态：

`剧本理解 -> 镜头准备 -> 资产确认 -> 镜头就绪 -> 图像/视频生成 -> 任务追踪 -> 结果回写`

《边界回声》共同驾驶舱已经有自己的剧本、宫格分镜、关键帧、Prompt 和 GitHub 审核流。因此更稳的路线是：保留现有文件型工作流，只吸收 Jellyfish 的“状态管理、资产一致性、准备度检查、任务追踪”思想。

## Jellyfish 可借鉴点

### 1. 资产实体独立管理

Jellyfish 把资产分为 Actor、Character、Scene、Prop、Costume。对《边界回声》的启发是：小夏、妈妈、澜冰屿、澪星应从普通 Prompt 描述升级为角色资产；儿童鞋、隔离门、旧北站候车区、废站长椅、澪星尾灯等应成为道具/场景资产。

### 2. 镜头准备状态

Jellyfish 的镜头不是只有“已生成/未生成”，而是区分基础信息是否完整、镜头语义是否完整、动作拍点是否存在、资产候选是否确认、对白候选是否确认、是否达到生成准备状态。

《边界回声》可以采用：

- `待拆解`
- `待关键帧`
- `待质检`
- `待视频`
- `视频待审`
- `通过`
- `需重做`
- `暂停`

### 3. 候选确认机制

Jellyfish 会把 AI 提取到的角色、场景、道具、对白作为候选项，让用户选择接受、忽略或链接到已有资产。

对《边界回声》来说，每次从剧本改分镜时，驾驶舱应自动列出本段涉及的资产锁：

- 小夏：确认使用 OP09 后右脚白袜、左脚单鞋。
- 妈妈：确认低发髻、浅卡其灰外套、危险侧。
- 隔离门：确认两米以上、高长连续、透明防爆板与金属框。
- 儿童鞋：确认 OP12-8 事故当天无人特写，OP13 才开始多年后腐蚀。

### 4. 视频准备度检查

Jellyfish 的视频生成前会检查镜头信息、时长、Prompt、参考帧、是否已有视频任务、模型和供应商配置。

共同驾驶舱可以做轻量版准备度：

| 检查项 | 规则 |
| --- | --- |
| 剧情锁 | 当前 OP 是否引用 ?? 最新剧本 |
| 连续性锁 | 角色、鞋子、场景时间线是否匹配 |
| 关键帧 | 是否满足每个 OP 6-8 张或宫格表要求 |
| Prompt | 是否有首帧、尾帧、动作拍点、负面约束 |
| 质检 | 是否达到 90 分以上 |
| 视频任务 | 是否已生成、待审或需要重做 |
| GitHub 审核 | 是否已进入共同驾驶舱 PR |

### 5. 任务中心

Jellyfish 用统一任务中心追踪文本、图片和视频任务。我们不需要马上做异步后台，但可以在管理器里先做“任务状态表”：

- 任务 ID。
- 所属 OP/宫格。
- 类型：关键帧、Prompt、视频、质检、修正。
- 状态：未开始、进行中、完成、失败、需人工审。
- 输入文件。
- 输出文件。
- 问题备注。
- 下一步动作。

## 不建议照搬的部分

### 1. 不建议现在部署完整 Jellyfish

Jellyfish 是 React + FastAPI + MySQL + Redis + 对象存储 + 异步任务系统。对当前《边界回声》来说，部署成本高于收益。

### 2. 不建议改掉文件型驾驶舱

现在的 Markdown、HTML 管理器、关键帧文件夹和 GitHub PR 审核已经适合你和 ChatGPT/Codex 协作。贸然改成数据库系统，反而会降低可读性。

### 3. 不建议让系统替代创作判断

Jellyfish 可以管理生产，但无法替代“妈妈松手是否被观众理解为保护”“OP12 静音落鞋是否足够痛”这类戏剧判断。共同驾驶舱应服务创作，不应把创作变成表格填空。

## 《边界回声》轻量升级路线

### 第一阶段：整理字段，不改架构

目标：1-2 天内可落地。

新增或整理以下文件：

- `production/cockpit/asset_registry.json`
- `production/cockpit/shot_registry.json`
- `production/cockpit/task_registry.json`
- `production/cockpit/readiness_rules.md`

用途：

- 资产表记录角色、场景、道具、服装锁。
- 镜头表记录每个 OP/宫格的状态、关键帧、Prompt、质检结果。
- 任务表记录下一步生成/审核工作。
- 准备度规则让驾驶舱能判断“这个镜头是否可进入视频生成”。

### 第二阶段：升级 AI 漫剧分镜管理器

目标：保留单 HTML 管理器，但增加生产面板。

新增视图：

- 资产库：角色、场景、道具、服装。
- 镜头准备度：每个 OP/宫格的红黄绿状态。
- 任务中心：当前待办、失败项、需人工审项。
- 一致性检查：鞋子、角色服装、场景时间线、隔离门规则。

优先顺序：

1. 镜头准备度。
2. 任务中心。
3. 资产库。
4. 一致性检查。

### 第三阶段：半自动生成报告

目标：让 Codex/ChatGPT 审核更快。

生成：

- 每日制作状态报告。
- 每个 OP 的关键帧联系表。
- 每个 OP 的视频准备度报告。
- GitHub PR 审核摘要。

## 对当前 OP10-OP12 的直接应用

当前已满足：

- OP10 有 7 张宫格关键帧。
- OP11 有 7 张宫格关键帧。
- OP12 有 8 张宫格关键帧。
- 质检综合 91 分可用。
- AI 漫剧分镜管理器已逐格引用。

仍需在视频准备度中标黄：

- OP10-4：保护性松手必须靠运动清楚表达。
- OP11-4：当前帧不是应急开关硬特写，必须靠视频运动补寻找门侧开关。
- OP12-7：当前帧偏崩溃停顿，必须靠视频运动补从停顿到奔跑。

## 最小可执行下一步

建议下一步不要立刻做复杂系统，而是先新增三个轻量文件：

1. `production/cockpit/asset_registry.json`
2. `production/cockpit/shot_registry_OP10_OP12.json`
3. `production/cockpit/readiness_rules.md`

然后把 AI 漫剧分镜管理器改成读取或内置这些字段，显示：

- 每个宫格的关键帧数。
- 每个宫格的准备度。
- 每个宫格的风险备注。
- 下一步视频任务。

这样就能把 Jellyfish 的核心优点移植过来，而不让整个项目变重。
